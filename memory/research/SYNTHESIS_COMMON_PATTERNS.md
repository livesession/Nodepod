## 1. The Universal Stack

### Every Tool Follows the Same Architecture

Every single Rust-to-WASM tool we have researched -- LightningCSS, Tailwind Oxide, Rolldown, and all napi-rs modules generally -- follows an identical technology stack. There are no exceptions. The stack is:

```
                      BUILD TIME                               RUNTIME
                      =========                               =======

    Rust Source Code                                    JavaScript Application
    (#[napi] annotated)                                        |
         |                                                     v
         v                                          .wasi.cjs / .wasi-browser.js
    napi-rs v3 crate                                  (generated JS glue code)
    (napi + napi-derive +                                      |
     napi-sys + napi-build)                                    v
         |                                         @napi-rs/wasm-runtime
         v                                         (re-exports emnapi + WASI)
    emnapi C library                                           |
    (libemnapi-basic-mt.a)                                     v
    [statically linked]                            @emnapi/core + @emnapi/runtime
         |                                         (N-API JS implementation)
         v                                                     |
    cargo build --target                                       v
    wasm32-wasip1-threads                          @emnapi/wasi-threads
         |                                         (ThreadManager, WASIThreads)
         v                                                     |
    .wasm binary                                               v
    (exports: napi_register_wasm_v1,               @tybys/wasm-util
     __napi_register__*, malloc,                   (WASI preview1 polyfill)
     free, wasi_thread_start)                                  |
                                                               v
                                                  WebAssembly.instantiate()
                                                  (SharedArrayBuffer memory)
```

### The Universal Equation

```
Rust Code + napi-rs --> wasm32-wasip1-threads --> .wasm binary
    +
emnapi (statically linked C) + @emnapi/core (JS) --> Node-API bridge
    +
@emnapi/wasi-threads (JS) --> Thread management
    +
SharedArrayBuffer + Web Workers --> Multi-threading
    =
Native-equivalent Rust module running in JavaScript
```

### Key Insight: There Is Only One Path

Unlike what one might expect, there is NOT a diversity of approaches. Every tool:
- Targets `wasm32-wasip1-threads` (not `wasm32-unknown-unknown`)
- Uses napi-rs v3 (not wasm-bindgen)
- Uses emnapi for the Node-API bridge (not a custom bridge)
- Uses `@emnapi/wasi-threads` for thread management (not wasm-bindgen-rayon in production)
- Uses SharedArrayBuffer for memory sharing (no alternative exists)
- Uses the same JS glue code pattern generated by `@napi-rs/cli`

The wasm-bindgen-rayon approach exists as an alternative for pure rayon workloads, but it uses `wasm32-unknown-unknown`, requires nightly Rust, and is NOT used by any of the tools we need to support.

## 2. The napi-rs Pattern

### Every Tool Follows This Exact Sequence

#### Phase 1: Build (Identical for ALL tools)

```
Rust source with #[napi] annotations
    |
    +--> napi-derive proc macro generates:
    |    - #[no_mangle] extern "C" fn __napi_register__<name>_<idx>()
    |      (WASM exports that populate static registries)
    |    - __napi_cb__<name> callback functions
    |      (actual function implementations)
    |
    +--> napi-build sets linker flags:
    |    --import-memory           (JS provides SharedArrayBuffer memory)
    |    --max-memory=4294967296   (4GB max)
    |    --export=malloc
    |    --export=free
    |    --export=napi_register_wasm_v1
    |    --export-table
    |    -zstack-size=64000000     (64MB stack)
    |
    +--> Links libemnapi-basic-mt.a (statically)
    |
    +--> cargo build --target wasm32-wasip1-threads
         Produces: <name>.wasm32-wasi.wasm
```

#### Phase 2: Load (Identical JS glue for ALL tools)

```javascript
// 1. Create shared memory
const memory = new WebAssembly.Memory({
    initial: N,       // Tool-specific (4000 for rolldown, 16384 for tailwind)
    maximum: 65536,   // 4GB
    shared: true      // ALWAYS true -- SharedArrayBuffer required
});

// 2. Create WASI instance
const wasi = new WASI({
    version: 'preview1',
    env: process.env,
    preopens: { [rootDir]: rootDir }
});

// 3. Instantiate via emnapi
const { instance, module, napiModule } = instantiateNapiModuleSync(wasmBytes, {
    context: getDefaultContext(),
    asyncWorkPoolSize: 4,
    reuseWorker: true,
    wasi,
    onCreateWorker() { /* create Worker */ },
    overwriteImports(importObject) {
        importObject.env = {
            ...importObject.env,
            ...importObject.napi,
            ...importObject.emnapi,
            memory
        };
    },
    beforeInit({ instance }) {
        for (const name of Object.keys(instance.exports)) {
            if (name.startsWith('__napi_register__')) {
                instance.exports[name]();  // Phase 3: Register
            }
        }
    }
});
```

#### Phase 3: Register (Automatic, same for ALL tools)

```
beforeInit callback:
  1. Iterate instance.exports
  2. Call every __napi_register__<name>_<idx>() export
  3. Each call populates MODULE_REGISTER_CALLBACK / MODULE_CLASS_PROPERTIES static registries

Then emnapi calls:
  4. napi_register_wasm_v1(env, exports)
  5. Reads registries, creates JS functions/classes for each registered item
  6. napiModule.exports now has all Rust-exported APIs
```

#### Phase 4: Thread (On-demand, same mechanism for ALL tools)

```
Rust: std::thread::spawn(closure)
  |
  v
wasi-libc: pthread_create()
  |
  v
WASI import: wasi_thread_spawn(start_arg)
  |
  v
JS: ThreadManager.threadSpawn()
  |
  +--> Get or create Worker (via onCreateWorker factory)
  +--> Post 'load' message: { wasmModule, wasmMemory }
  +--> Worker instantiates SAME module with SAME shared memory
  +--> Worker posts 'loaded' response
  +--> Post 'start' message: { tid, startArg }
  +--> Worker calls instance.exports.wasi_thread_start(tid, startArg)
  +--> Thread runs Rust closure
  +--> On completion: 'cleanup-thread' message
  +--> Worker returned to pool (if reuseWorker)
```

### What Makes This Pattern Universal

The reason every tool uses this exact pattern is that napi-rs v3's WASM support is the only production-quality path for Rust-to-WASM with Node-API compatibility. The alternatives:
- **wasm-bindgen**: Different ABI, requires nightly for threads, no napi compatibility
- **wasm-bindgen-rayon**: Only handles rayon, not general threading or napi
- **Raw WASI**: No Node-API bridge -- you would need to build one from scratch

## 4. The Threading Model

### The Unified Threading Architecture

All threading in Rust-to-WASM follows the **instance-per-thread** model:

```
Main Thread                    Worker Thread 1              Worker Thread 2
+------------------+          +------------------+         +------------------+
| WASM Instance 0  |          | WASM Instance 1  |         | WASM Instance 2  |
| - Own globals    |          | - Own globals    |         | - Own globals    |
| - Own TLS        |          | - Own TLS        |         | - Own TLS        |
| - Own func table |          | - Own func table |         | - Own func table |
|    +-------------+-----------+--------+--------+---------+                  |
|    |                SHARED LINEAR MEMORY                 |                  |
|    |    (WebAssembly.Memory with shared: true)           |                  |
|    |    heap, stacks, atomics, synchronization data      |                  |
|    +-----------------------------------------------------+                  |
+------------------+                                       +------------------+
```

Each thread gets:
- **Own WASM instance**: separate globals, function table, call stack
- **Own TLS**: `__tls_base` global, initialized via `__wasm_init_tls(ptr)`

All threads share:
- **Linear memory**: backed by SharedArrayBuffer
- **Heap**: malloc/free operate on shared memory
- **Atomics**: `i32.atomic.*` instructions for synchronization

### How std::thread::spawn Maps to Workers

```
    Rust                    wasi-libc                 WASI Host (JS)            Worker
    ====                    =========                 ==============            ======
std::thread::spawn(f)
    |
    +-> pthread_create()
         |
         +-> Allocate stack in shared memory
         +-> Allocate TLS block
         +-> wasi_thread_spawn(start_arg) -----> ThreadManager.threadSpawn()
                                                      |
                                                      +-> getNewWorker()
                                                      |   (pool or onCreateWorker)
                                                      +-> post 'load' { module, memory }
                                                      |                        |
                                                      |              instantiate WASM
                                                      |              with shared memory
                                                      |              post 'loaded'
                                                      +-> post 'start' { tid, arg }
                                                                               |
                                                                    __wasm_init_tls(tls_block)
                                                                    wasi_thread_start(tid, arg)
                                                                    --> runs closure f
                                                                    --> cleanup-thread
    <-- return tid (or error) <-----------------------------
```

### SharedArrayBuffer: The Backbone

SharedArrayBuffer is the single most critical dependency. Without it, NOTHING works:

| What It Enables | Mechanism |
|-----------------|-----------|
| Shared WASM memory | `WebAssembly.Memory({ shared: true })` |
| Mutex / Condvar | `memory.atomic.wait32` / `memory.atomic.notify` |
| Work-stealing deques (rayon) | Atomic CAS on shared deque pointers |
| Thread signaling (latches) | `AtomicUsize` state machines |
| FS proxy RPC | `Atomics.wait(sab)` / `Atomics.notify(sab)` |
| Tokio parking | `Condvar::wait()` --> `memory.atomic.wait32` |
| Sleep system (rayon) | Progressive backoff --> `condvar.wait()` |

### The emnapi ThreadManager Orchestration

```typescript
class ThreadManager {
    unusedWorkers: WorkerLike[] = [];     // Pre-warmed idle workers
    runningWorkers: WorkerLike[] = [];    // Currently executing threads
    pthreads: Record<number, WorkerLike> = {};  // TID -> Worker mapping

    wasmModule: WebAssembly.Module;       // Shared across ALL workers
    wasmMemory: WebAssembly.Memory;       // SharedArrayBuffer-backed

    // Thread lifecycle:
    getNewWorker() -> WorkerLike         // Pool or factory
    loadWasmModuleToWorker(worker)       // Post module + memory
    markId(worker) -> number             // TID = nextWorkerID + 43
    returnWorkerToPool(worker)           // Reuse after thread exit
}
```

Message protocol between main thread and workers:

| Message | Direction | Purpose |
|---------|-----------|---------|
| `{ __emnapi__: { type: 'load', payload: { wasmModule, wasmMemory } } }` | Main -> Worker | Initialize worker |
| `{ __emnapi__: { type: 'loaded' } }` | Worker -> Main | Worker ready |
| `{ __emnapi__: { type: 'start', payload: { tid, arg, sab } } }` | Main -> Worker | Start thread |
| `{ __emnapi__: { type: 'cleanup-thread', payload: { tid } } }` | Worker -> Main | Thread finished |
| `{ __emnapi__: { type: 'spawn-thread', payload: { startArg } } }` | Worker -> Main | Sub-thread spawn |
| `{ __fs__: { sab, type, payload } }` | Worker <-> Main | FS proxy RPC |

### When Rayon vs Tokio vs Both

| Tool | Rayon | Tokio | How They Interact |
|------|-------|-------|-------------------|
| **LightningCSS** | No | No | Single-threaded napi-rs only |
| **Tailwind Oxide** | Yes | No | Rayon `par_iter()` for parallel file scanning + extraction |
| **Rolldown** | Yes | Yes | Tokio for async I/O (module loading), Rayon for CPU-bound (chunk rendering) |

When BOTH are used (Rolldown):
- **Tokio** runs on the current-thread scheduler (or multi-thread with `tokio_unstable`)
- **Rayon** has its own thread pool (separate from tokio's blocking pool)
- Both share the same `WebAssembly.Memory` and use the same `wasi_thread_spawn` mechanism
- Both thread pools are managed by `@emnapi/wasi-threads` ThreadManager

Thread pools are configured independently:
- Rayon: `RAYON_NUM_THREADS` env var or `thread::available_parallelism()`
- Tokio blocking pool: internal management
- emnapi async work: `asyncWorkPoolSize` (default 4) or `NAPI_RS_ASYNC_WORK_POOL_SIZE` env var

## 5. The FS Access Pattern

### Three Tiers of Filesystem Access

#### Tier 1: WASI fd_* Operations (Native-Like FS)

Used by ALL tools when Rust code calls `std::fs::*`:

```
Rust: std::fs::read_to_string("file.js")
  |
  v
wasi-libc: open() + read() + close()
  |
  v
WASI preview1 imports:
  path_open(dirfd, dirflags, path, ...) -> fd
  fd_read(fd, iovs, iovs_len) -> nread
  fd_close(fd)
  fd_seek(fd, offset, whence) -> newoffset
  fd_filestat_get(fd) -> filestat
  fd_readdir(fd, buf, buf_len, cookie) -> bufused
  path_filestat_get(fd, flags, path) -> filestat
```

These operations require:
- **Preopened directories**: The WASI instance maps host directories to WASI fd numbers
- **Node.js path**: Uses `node:wasi` which delegates to real `node:fs`
- **Browser path**: Uses `@tybys/wasm-util` WASI which delegates to `memfs`

#### Tier 2: FS Proxy via SharedArrayBuffer RPC

Used by WASM worker threads that cannot directly access `node:fs`:

```
Worker Thread                              Main Thread
    |                                           |
    +-- postMessage({                           |
    |     __fs__: {                             |
    |       sab: new Int32Array(SAB),           |
    |       type: 'readFileSync',               |
    |       payload: ['/project/file.js']       |
    |     }                                     |
    |   })                                      |
    |                                           +-- fs.readFileSync('/project/file.js')
    +-- Atomics.wait(sab, 0, 21)               +-- encode result to SAB
    |   [BLOCKED]                               +-- Atomics.store(sab, 0, 0)  // success
    |                                           +-- Atomics.notify(sab, 0)
    +-- [WAKES UP]                              |
    +-- decode result from SAB                  |
    +-- return value                            |
```

SAB layout:
```
Bytes 0-3:    status (int32)  -- 21=waiting, 0=success, 1=error
Bytes 4-7:    type (int32)    -- 0=undefined, 1=null, 2=boolean, 3=number, 4=string, 6=json, 9=bigint
Bytes 8-15:   payload_size (uint32)
Bytes 16+:    payload content (max 10,240 bytes)
```

**Critical limitation:** 10,240 byte payload maximum. Large file reads will fail.

#### Tier 3: memfs (Browser Alternative)

When real filesystem is unavailable (browser environments):

```javascript
import { memfs } from '@napi-rs/wasm-runtime/fs'
const { fs, vol } = memfs()

// Populate memfs with project files before scanning
vol.fromJSON({ '/src/App.tsx': '...', '/src/index.css': '...' })

// WASI uses memfs instead of real fs
const wasi = new WASI({ fs, preopens: { '/': '/' } })
```

### How Content Scanning Differs from Module Resolution

| Aspect | Tailwind Oxide (scanning) | Rolldown (bundling) |
|--------|---------------------------|---------------------|
| **Pattern** | Read thousands of files, extract class names | Read files, parse AST, resolve imports, transform |
| **FS operations** | `readdir` + `read` (bulk parallel) | `open` + `read` + `stat` (sequential chains) |
| **Parallelism** | Rayon `par_iter()` over all files | Tokio async I/O for concurrent module loading |
| **File sizes** | Small-medium (HTML, JSX) | Any size (JS, TS, CSS, assets) |
| **Write** | Never | Yes (output bundles) |
| **Watch mode** | `stat()` for mtime tracking | Full re-resolve cycle |
| **Symlinks** | Follows (`follow_links(true)`) | Resolves canonically |
| **gitignore** | Respects (custom `ignore` crate) | Does not apply |

### For nodepod Specifically

nodepod operates as a browser-based runtime with a Virtual File System (VFS). The mapping is:

```
WASI fd_* operations  -->  nodepod VFS (fsBridge)
FS proxy from workers -->  nodepod VFS (via createOnMessage(fsBridge))
memfs (browser path)  -->  patched to return nodepod VFS
```

Key: nodepod patches `@napi-rs/wasm-runtime/fs`'s `memfs()` to return `{ fs: fsBridge, vol: volume }` so that WASI file operations route through the VFS.

## 7. Common Gotchas

### Gotchas Shared Across ALL Tools

#### 1. Import Namespace Merging (CRITICAL)

emnapi provides imports in three separate namespaces (`napi`, `emnapi`, `env`), but the WASM module expects ALL of them in the flat `env` namespace. The `overwriteImports` callback MUST merge them:

```javascript
importObject.env = {
    ...importObject.env,      // WASI standard env imports
    ...importObject.napi,     // N-API functions
    ...importObject.emnapi,   // emnapi internals
    memory: sharedMemory      // The shared memory object
};
```

If your runtime provides imports differently (e.g., separate namespaces), this merge will fail silently and the WASM module will trap on first N-API call.

#### 2. Registration Ordering (CRITICAL)

The `__napi_register__*` functions MUST be called BEFORE `napi_register_wasm_v1`. They populate static registries (`MODULE_REGISTER_CALLBACK`, `MODULE_CLASS_PROPERTIES`) that the registration function reads. If called out of order, the module exports will be empty.

#### 3. SharedArrayBuffer Availability

Without SharedArrayBuffer, the `WebAssembly.Memory({ shared: true })` constructor will throw. This is the FIRST thing that fails. Ensure:
- Node.js: works by default
- Browsers: requires COOP/COEP headers
- Custom runtimes: must implement SharedArrayBuffer

#### 4. Main Thread Cannot Block (Browser Only)

`Atomics.wait()` throws `TypeError` on the browser main thread. This affects:
- Rayon's `Mutex::lock()` / `Condvar::wait()` (backed by `memory.atomic.wait32`)
- Tokio's parking mechanism
- FS proxy RPC

**Solution:** In Node.js/nodepod, `Atomics.wait()` works on ALL threads including main. This gotcha is browser-specific. However, rayon provides `web_spin_lock` feature as a workaround, and the standard practice is to run WASM in a Worker rather than the main thread.

#### 5. TLS Template Contamination

The `_initialize()` export writes the main thread's state into the TLS template (in BSS). Child threads copy this contaminated template and may panic with:
- "capacity overflow"
- "thread handle already set during thread spawn"
- "memory allocation of N bytes failed"

This was the root cause of nodepod's threading issues. Workaround: limit threads (`RAYON_NUM_THREADS=1`).

#### 6. FS Proxy Payload Limit

The SharedArrayBuffer-based FS proxy has a **10,240 byte maximum payload**. Large file reads fail with `RangeError: payload overflow`. This affects:
- Tailwind Oxide scanning large source files
- Rolldown reading large JavaScript files
- Any `readFileSync` call returning > 10KB

**Mitigation:** The FS proxy is only used by worker threads. The main thread's WASI instance reads files directly through preopens.

#### 7. Worker Creation Latency

Worker creation costs 50-200ms each. For rayon workloads with many short tasks, this is devastating. Always configure `reuseWorker: true` and pre-warm the pool.

#### 8. The Worker `unref` Hack

The generated glue code uses `Object.getOwnPropertySymbols(worker)` to find Node.js internal `kPublicPort` and `kHandle` symbols and neuter their `ref()` methods. This is Node.js-specific and will not work in other runtimes. Custom runtimes must implement their own worker lifecycle management.

#### 9. Memory Limits Must Be Pre-Declared

Shared `WebAssembly.Memory` requires a `maximum` page count at creation time. It cannot be changed later. Typical values:
- LightningCSS: ~256MB initial
- Tailwind Oxide: 1GB initial, 4GB max
- Rolldown: ~256MB initial, 4GB max

Under-provisioning causes OOM. Over-provisioning wastes address space.

#### 10. The `tokio_unstable` Flag

Without `RUSTFLAGS='--cfg tokio_unstable'`, tokio on WASM:
- Uses single-thread runtime only
- Spawns real OS threads (via `std::thread::spawn`) for async work
- Cannot use `rt-multi-thread` feature

With `tokio_unstable`:
- Can use multi-thread runtime
- Async futures use `tokio::spawn` instead of `std::thread::spawn`

Rolldown is built with `tokio_unstable` in CI.

## 8. A Unified Loading Architecture

### Single Architecture for ANY napi-rs WASM Module in nodepod

```
nodepod Script Engine
    |
    v
Module Resolution
    |
    +--> Native .node binary? --> FAIL (not available in browser)
    |
    +--> Platform-specific package? --> FAIL (e.g., @pkg/binding-linux-x64-gnu)
    |
    +--> NAPI_RS_FORCE_WASI or fallback
         |
         v
    WASM Package Resolution
    (try .wasi.cjs local, then @pkg/binding-wasm32-wasi)
         |
         v
    NapiWasmLoader (unified loader)
         |
         +---------------------------------------------------+
         |                                                     |
    1. Read .wasm binary from VFS                              |
         |                                                     |
    2. Create shared memory:                                   |

         initial: config.initialMemory || 4000,                |
         maximum: config.maximumMemory || 65536,               |
         shared: true                                          |
       })                                                      |
         |                                                     |
    3. Create WASI instance:                                   |
       wasi = new WASI({                                       |
         version: 'preview1',                                  |
         env: process.env,                                     |
         preopens: { '/': '/' },  // root maps to VFS         |
         fs: fsBridge             // nodepod VFS               |
       })                                                      |
         |                                                     |
    4. Instantiate with emnapi:                                |
       instantiateNapiModuleSync(wasmBytes, {                  |
         context: getDefaultContext(),                          |

       })                                                      |
         |                                                     |
    5. Return napiModule.exports                               |
         +---------------------------------------------------+
```

### The `createNodepodWorker` Factory

```javascript
function createNodepodWorker(config) {
    // 1. Read worker script from VFS (wasi-worker.mjs or wasi-worker-browser.mjs)
    // 2. Bundle it with dependencies (buildWorkerBundle)
    // 3. Create blob URL worker (classic, not module)
    // 4. Add Node.js Worker API surface (.on, .off, .emit, .unref, .ref)
    // 5. Attach FS proxy handler (createOnMessage with fsBridge)
    // 6. Track ref count for event loop
    return patchedWorker;
}
```

### The Common Initialization Sequence

```
Time  Action                                              Who
----  ------                                              ---
 0    Read .wasm from VFS                                 nodepod
 1    Create WebAssembly.Memory({ shared: true })         nodepod
 2    Create WASI instance with VFS preopens              nodepod
 3    Build import object (WASI + emnapi + napi)          @emnapi/core
 4    WebAssembly.instantiate(bytes, imports)              V8
 5    WASIThreads.initialize(instance, module, memory)    @emnapi/wasi-threads
 6    Pre-load worker pool (N workers)                    @emnapi/wasi-threads
 7    Each worker: instantiate WASM with shared memory    Workers
 8    beforeInit: call all __napi_register__* exports     nodepod
 9    napi_register_wasm_v1(env, exports)                 emnapi
10    napiModule.exports ready                            emnapi
----  [Module is now usable]                              ---
11+   On demand: std::thread::spawn -> wasi_thread_spawn  Rust
      -> ThreadManager.threadSpawn -> Worker              @emnapi/wasi-threads
12+   On demand: FS access from worker -> FS proxy RPC   @napi-rs/wasm-runtime
```

This sequence is IDENTICAL for LightningCSS, Tailwind Oxide, Rolldown, and any future napi-rs WASM module.

## 9. Performance Characteristics

### CPU-Bound vs I/O-Bound Profile

| Tool | Profile | Dominant Operation | Parallelism Style |
|------|---------|-------------------|-------------------|
| **LightningCSS** | CPU-bound | CSS parsing + transformation | None (single-threaded) |
| **Tailwind Oxide** | Mixed (CPU + I/O) | File scanning + candidate extraction | Rayon `par_iter()` (data parallel) |
| **Rolldown** | Mixed (I/O + CPU) | Module loading (I/O) + chunk rendering (CPU) | Tokio (async I/O) + Rayon (CPU parallel) |

### Threading Needs

| Tool | Thread Count | Pool Type | Thread Lifetime |
|------|-------------|-----------|-----------------|
| **LightningCSS** | 0 extra | None | N/A |
| **Tailwind Oxide** | 4 (default `asyncWorkPoolSize`) | Rayon work-stealing + emnapi async | Long-lived (pool reuse) |
| **Rolldown** | 4+ emnapi + N rayon + M tokio | Three separate pools | Long-lived (pool reuse) |

### Memory Usage

| Tool | Initial Memory | Typical Usage | Maximum |
|------|---------------|---------------|---------|
| **LightningCSS** | ~256MB (4000 pages) | Low (~50-100MB) | 4GB |
| **Tailwind Oxide** | 1GB (16384 pages) | Moderate (~200-500MB for large projects) | 4GB |
| **Rolldown** | ~256MB (4000 pages) | High (~500MB-2GB for large bundles) | 4GB |

### WASM vs Native Performance

| Tool | WASM Overhead | Primary Bottleneck |
|------|--------------|-------------------|
| **LightningCSS** | ~2-3x slower | CPU (parsing) |
| **Tailwind Oxide** | ~2-5x slower | I/O (file reading) + CPU (extraction) |
| **Rolldown** | ~3-5x slower | I/O (module resolution) + CPU (rendering) |

The overhead comes from:
1. WASM instruction execution (~1.5-2x slower than native)
2. JS/WASM boundary crossing (N-API calls through emnapi)
3. Thread creation latency (Worker spawn vs OS thread)
4. FS proxy overhead (SharedArrayBuffer RPC for worker FS access)
5. Memory management (shared memory atomics vs native locks)

### Startup Latency

| Phase | Time | Applies To |
|-------|------|-----------|
| WASM compilation | 100-500ms | All tools |
| WASM instantiation | 50-200ms | All tools |
| Worker pool pre-warm | 50-200ms per worker | Threaded tools |
| napi registration | 5-20ms | All tools |
| **Total cold start** | **200-1200ms** | Depends on tool |

### Optimization Opportunities

1. **Worker pool pre-warming**: Start workers before first use
2. **WASM caching**: Cache compiled `WebAssembly.Module` across loads
3. **Thread limiting**: `RAYON_NUM_THREADS=1` avoids threading overhead when not needed
4. **Memory right-sizing**: Don't over-allocate initial memory
5. **FS proxy bypass**: Route main-thread WASI operations directly through VFS

## Summary

### The Single Most Important Insight

All six technologies converge on a single implementation path:

```
napi-rs v3 + wasm32-wasip1-threads + emnapi + SharedArrayBuffer + Web Workers
```

There is no diversity of approaches. Every tool uses the same stack, the same glue code pattern, the same threading model, and the same set of JS dependencies. This means nodepod needs to implement ONE coherent loading architecture that supports this stack, and ALL current and future napi-rs WASM tools will work.

### The Three Levels of Support

1. **Level 1 (LightningCSS):** Basic napi-rs WASM loading. No threading needed beyond the emnapi async pool. Requires: SharedArrayBuffer, WASI preview1, emnapi imports, correct registration ordering.

2. **Level 2 (Tailwind Oxide):** Level 1 + Rayon parallelism. Requires: Worker thread creation, thread pool management, extensive WASI filesystem operations (directory walking, file reading, metadata).

3. **Level 3 (Rolldown):** Level 2 + Tokio async runtime. Requires: `clock_time_get` for timers, `tokio_unstable` build flag, potentially `poll_oneoff` for I/O driver, concurrent rayon + tokio thread pools.

nodepod has already achieved Level 2 (with thread limiting as a workaround for TLS contamination). Full Level 3 support is the target.
